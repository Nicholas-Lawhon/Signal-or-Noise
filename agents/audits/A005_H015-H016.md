## Findings

- **[MAJOR-1] Amazon scenario title "Peak Expectations" is a doc 09 named title-bias FAIL example.**
  `scenario_amazon_1999_2001.json` uses the exact title doc 09's Directional
  Sentiment Rules cite as pre-answering the call ("titles such as 'Defensive
  Compounder' or 'Peak Expectations' pre-answer the call"), and the outcome is
  indeed a −87% short win. Directional sentiment is warn-tier, not a literal
  identity leak, so per H017 this does not fail the audit for prototype seeds —
  but an active seed should not ship a rulebook-named FAIL example.
- **[MAJOR-2] Pre-decision-visible metadata fields are outside the leakage scan.**
  Classic Run displays `scenario.era`, `decisionDateLabel`, and
  `holdingPeriodLabel` before the decision (`run/page.tsx:155-160`), but
  `collectHiddenTexts` scans only title + the six variant fields + setupHints.
  Probe confirmed: a scenario with `era: "The Netflix Streaming Era"` or
  `decisionDateLabel: "Netflix Jan 2012"` passes validation. This matches the
  H015 spec's hidden-text definition (so it is not an acceptance failure) and
  all six current seeds are clean, but it is a real anti-leak gap in the
  production validation domain (D019 intent: reject before a human reviews).
- **[MAJOR-3] The literal-leak layer is only as strong as each scenario's authored `identityBannedTerms`.**
  `checkLeakage` scans company name, ticker, accepted names, and
  `identityBannedTerms` (`validation.ts:83-112`), but `identityBannedTerms`,
  `factBank` lists, and likely-guess arrays may be empty by schema
  (`schema.ts:48,98-101`). Acceptance passes because the active samples include
  useful banned terms and the required tests pass, but omitted founder/product/
  slogan terms silently bypass the automated D019 layer for future content.
- **[MINOR-1] ISO date schema accepts impossible calendar dates.** The regex
  `^\d{4}-\d{2}-\d{2}$` admits `2009-13-45` (probe confirmed validation passes).
  Lexicographic comparison is correct for well-formed zero-padded dates, so the
  window checks themselves are sound; a calendar-validity `.refine` would close it.
- **[MINOR-2] No internal price/return consistency checks.**
  `actualReturnPercent` vs `startingPrice`/`endingPrice`, and lookback-last vs
  outcome-first continuity, are unchecked (current seeds happen to be
  consistent, e.g. Netflix 10.3 → 10.3). Not required by H015.
- **[MINOR-3] No Medium setup-hint overflow test.** The 0–1 rule is enforced in
  code but no test feeds Medium 2 hints; H015 only required the Easy-0/Hard-1 cases.
- **[MINOR-4] `hardLikelyGuesses` are placeholders, not guess lists.** All six
  seeds record e.g. `["semiconductor peers"]` instead of ≥4 named companies as
  doc 09 Gate 1 requires. Acceptable for prototype seeds but the field as used
  gives no red-team value.

## Validator Adversarial Review

- **Hidden-text coverage:** `collectHiddenTexts` (`validation.ts:49-81`)
  includes `scenario.title`, all three variants' `companyDescription`,
  `macroContext`, `situation`, `longCase`, `shortCase`, and every `setupHints`
  entry — matches the H015/H017 field list. (Gap for non-hidden pre-decision
  metadata is MAJOR-2.)
- **Reveal/company metadata exempt:** confirmed — only title + hiddenCard are
  scanned; `reveal.*`, `company.*`, `review.*` may name the company (all six
  seeds do, correctly — e.g. "That was Netflix" — and still validate).
- **Return decimal guard:** probe confirmed `11.356` and `19.9` pass, `20` and
  `1135.6` fail (`abs >= 20`). Matches the H015 conservative-guard spec exactly,
  including the boundary.
- **Date comparisons:** lexicographic comparison of schema-constrained
  zero-padded `YYYY-MM-DD` strings is valid; all five H015 window rules are
  implemented (pre-start ≤ pre-end ≤ decision; outcome-start ≥ decision;
  outcome-end ≥ outcome-start; decision < end). Impossible-calendar-date gap is
  MINOR-1.
- **Warnings never fail:** confirmed in code (`success: true` with warnings
  when no errors) and by the passing warning test; the CLI prints warnings and
  exits 0 for them.
- **Term-matching robustness:** regex-special characters escaped; multi-word
  banned terms match across whitespace; word boundaries catch possessives
  ("Netflix's" fails validation — probe confirmed) without false positives on
  embedded substrings ("supervisable" passes). Single-letter ticker `V` is
  handled by the same word-boundary logic and the Visa seed passes.
- **Structural rules:** all three variants required by schema plus
  `difficultySupported` must include all three (business rule); empty
  `lookbackPrices`/`outcomePrices` rejected via `.min(1)`; sources `.min(1)`
  with URL validation; `whyItMoved` is an exact 3-tuple.
- **No BLOCKER-tier blind spots found:** every check H015/D019/D026 required is
  present and behaves as specified.
